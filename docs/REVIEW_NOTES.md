# プロジェクトレビューメモ（2026-04-03）

## 概要

プロジェクトの目的「音響プログラミング初心者が、Google Colabのノートブックを通じて、デジタル信号処理と音響合成の基礎を学ぶ」に対して、仕様が適切かをレビューした結果のメモ。

## 評価

### 良い点

- **カリキュラム設計が段階的で適切** — サイン波→エンベロープ→フィルター→エフェクト→シーケンサー→分析→最終制作という流れは王道の順序で、依存関係も正しい
- **モジュール分割が教育目的に合致** — `core`/`synthesis`/`effects`/`instruments`/`sequencer`の分割がレッスンの進行と対応
- **「ブラックボックスの排除」が一貫** — 純Python + NumPyで実装し、数式がそのままコードになる設計哲学が明確
- **Colab前提の設計** — 環境構築の障壁を下げ、セットアップガイドやアシスタント向けチェックリストも整備

## 懸念点・改善提案

### 1. スコープが目的に対して広すぎる

- **Lesson 05（MIDI・シーケンサー）** が「理論と実装の橋渡し」から逸れ、音楽制作ツール寄りになっている。MIDIノート番号と周波数の変換は有用だが、マルチトラックシーケンサーの自作まで必要か疑問
- **Lesson 07（最終プロジェクト）** の「3〜5分のオリジナル楽曲制作」は音響プログラミング学習の到達点として飛躍がある。信号処理の理解を確認する課題（特定の音色の再現、フィルタの設計課題等）の方が目的に沿う
- **instruments/** モジュールは独立モジュールとして提供するほどではない可能性がある

### 2. Lesson 06（サンプリングと分析）の位置が不自然

サンプリング定理やFFTは音響プログラミングの**基礎理論**であり、Lesson 01〜02あたりで扱うべき内容。現在の配置だと、Lesson 01〜05で「なぜサンプリングレートが44100なのか」「周波数とは何か」を理論的裏付けなしに使い続けることになる。

**提案**: サンプリング定理の基本をLesson 01に統合し、FFT/スペクトログラムをLesson 03（フィルター）の前に持ってくると、フィルターが「何をしているか」を周波数領域で理解できる。

### 3. Python 3.8サポートは不要

- Google Colabは現在Python 3.10+
- Python 3.8は2024年10月にEOL済み
- 教育用途であれば最新環境前提で問題ない
- 3.8サポートは型ヒントの書き方などに不要な制約を課す

**提案**: `>=3.10` に引き上げ。

### 4. ドキュメント群が過剰

10個のドキュメント（CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, CHANGELOG, 2つのColab guide, GitHub guide, Release checklist, Test report, Assistant checklist）は個人〜小規模の教育プロジェクトには過剰。

- SECURITY.md（「24-72時間の対応」）は教育用ライブラリには不要
- GITHUB_RELEASE_CHECKLISTやGITHUB_SETUP_GUIDEは運用メモでありリポジトリに含める必要がない
- CHANGELOG（v1.0.0のみ）は現段階では不要

**提案**: README, CONTRIBUTING（簡潔版）, COLAB_SETUP_GUIDE, ASSISTANT_CHECKLIST の4つ程度に削減。

### 5. テスト・CIの構成が目的と乖離

- CIが3つ（`test.yml`, `ci-lite.yml`, `ci.yml`）あるのは過剰。教育用ライブラリには1つで十分
- `mypy --strict` は教育的なコードの可読性を損なう可能性がある（型注釈が複雑になりがち）
- テストマーカーが8種あるがテストファイルは3つしかない

**提案**: CIを1つに統合。

### 6. READMEが冗長

READMEが567行あり、同じコード例が繰り返し登場（基本的な使用方法が3回以上）。「既存ライブラリとの比較」セクションはライブラリの利用者（=学生）にとっては不要で、教員向けの別ドキュメントに分離すべき。

**提案**: 学生に必要な情報（セットアップ方法、レッスンの始め方、API一覧）に絞る。

## リファクタリング計画

### 背景

レビューの結果、`AudioConfig`を各所に渡す現行設計に根本的な問題があることが判明。
`AudioConfig`は`sample_rate`, `bit_depth`, `max_amplitude`を混在して持つが、
実際には`sample_rate`だけが音響処理全体に必要で、他はWAV保存の実装詳細。
音響の世界では「sample_rateはデータと不可分」が主流（librosa, scipy, pydub等）であり、
信号自体にsample_rateを持たせる`AudioSignal`クラスの導入が教育的にも設計的にも適切と判断。

### 進め方

API変更がライブラリ全体に波及するため、他の改善項目も含めて段階的に実施する。

#### Phase 1: AudioSignal仕様設計 ✅ 完了
- AudioSignalクラスのインターフェース設計 → `docs/DESIGN_AUDIO_SIGNAL.md`
- 既存APIからの変更点と影響範囲の洗い出し
- AudioConfig廃止に伴う移動先: `bit_depth`→save()内部定数(16bit固定), `max_amplitude`→save()内部定数(0.95固定)
- 波形生成/エンベロープは関数化、フィルター/エフェクト/楽器/シーケンサーはオブジェクト維持

#### Phase 2: ライブラリ本体の実装 ✅ 完了
- AudioSignalクラスの実装
- 既存クラス（oscillators, envelopes, effects, instruments, sequencer）のAPI変更
- wave_io（save/load）のAPI変更
- テストの修正・既存3件の失敗解消

#### Phase 3: 周辺の整理 ✅ 完了
- ドキュメント削減（10個→4個: README, CONTRIBUTING簡潔版, COLAB_SETUP_GUIDE, ASSISTANT_CHECKLIST）
- CIを1つに統合（ci.yml, ci-lite.yml, test.yml → ci.yml）
- READMEを学生向けに簡潔化（セットアップ、レッスンの始め方、API一覧に絞る）
- ノートブック（colab_lessons/）を新APIに更新
- examples/を新APIに更新
